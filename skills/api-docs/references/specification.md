# API 文档编写规范

按本规范编写 API 文档，保证格式统一、可供第三方或前端对接使用。

## 文档整体结构

1. **文档标题**：一级标题，如 `# XXX系统API文档`
2. **目录**（可选）：`[TOC]`
3. **元信息**：用引用块写版本号、创建/更新时间、作者、密级、文档语法等
4. **修改记录**：`## 修改记录`，下接 `### 版本号` + 列表说明变更
5. **说明**：`## 说明`，简要说明文档用途与读者
6. **签名/认证**（按需）：如 `## 1. API签名方法`，含请求地址、公共参数表、签名算法、调用示例、注意事项
7. **错误码**：`## 2. 错误码`，公共错误码表、业务错误码表（表格）
8. **接口列表**：`## 3. 接口`，下接各接口小节

## 单个接口小节结构

- **标题**：`### 3.N. 接口名称(中文)`，如 `### 3.1. 创建订单(下单)`
- **请求行**：`` `METHOD` `` `` `路径` ``，如 `` `POST` `` `` `/api/vendor/orders/create/` ``
- **说明**（可选）：引用块 `> ...` 说明业务规则或注意点
- **上行参数**：
  - `request headers`参数：表格，列依次为「参数」「类型」「必填」「默认值」「备注」
  - `request json`参数：表格，列同上；嵌套字段用 `└` 表示层级（如 `└srv_order_no`）
- **下行参数**：表格，列依次为「参数」「类型」「默认值」「备注」；嵌套用 `└`
- **成功/失败示例**：`成功` 返回JSON示例、`失败` 返回JSON示例，各跟一个 ```json 代码块

## 表格与用词约定

- **上行参数**：指请求（headers + body/params）
- **下行参数**：指响应 body 结构说明
- 参数表表头使用：`参数`、`类型`、`必填`、`默认值`、`备注`（下行可无「必填」）
- 类型写 string/int/object/array 等，备注中枚举用 `KEY=说明` 分行列出
- 嵌套结构在「参数」列用 `└`、`└└` 表示层级

## 响应格式约定

- 成功：`code` 为 0（或文档约定的成功码），`msg` 如 "SUCCESS"，`data` 为业务数据
- 失败：`code` 非 0，`msg` 为错误描述；可无 `data`
- 示例中分别给出「成功」与「失败」的 JSON 示例

## Markdown 规范（必须遵守）

- 标题：ATX 风格（`#`），`#` 后有一个空格，标题前后空行，层级递增不跳级
- 列表：统一使用 `-`，符号后有一个空格，列表前后空行
- 代码块：围栏式，必须指定语言（`json`、`python`、`plain` 等），代码块前后空行
- 行尾无空格，无连续多空行，文件以空行结尾
- 文档以一级标题开始，全文仅一个一级标题

## 文件命名规范

- API 文档文件名格式：`XXXAPI文档.md`，如 `病历复印系统患者端API文档.md`、`病历复印系统管理端API文档.md`

## 禁止省略（必须遵守）

- **禁止引用其他接口代替描述**：不允许写「与XXX接口相同」「同XXX」「与患者端相同」等省略写法，每个接口必须独立完整地列出全部上行参数表、下行参数表和成功/失败JSON示例
- **禁止模糊参数说明**：不允许写「XXX等字段」「其他schema字段」「同上」「参数同创建」等省略用语，每个参数必须在表格中逐一列出，写明参数名、类型、是否必填、默认值和备注
- **枚举值必须列出**：如果参数的取值为枚举类型，必须在「备注」列中逐一列出所有可选值及其含义，格式为 `KEY=说明`，多个枚举值用 `<br>` 分行，例如：`MALE=男<br>FEMALE=女<br>UNKNOWN=未知`
