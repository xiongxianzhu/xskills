# 钉钉工作日志生成器

把用户随口说的「碎碎念」变成专业、得体的工作汇报语言，输出可直接粘贴到钉钉的编号列表。

## 何时使用

- 用户要写钉钉日志、工作日报、今日完成总结
- 用户提到「写日志」「日报」「钉钉日志」「今日工作」「工作汇报」「daily report」等

更多场景、输入输出示例见 [references/usage.md](references/usage.md)。

## 工作流程

### Step 1: 接收输入

用户以**换行逐条**的方式输入今天的任务，每行一条简短描述。例如：

```
修复登录bug
优化首页加载
对接支付接口
```

如果用户用逗号/顿号/空格在同一行分隔了多个任务，先拆分为独立条目。

### Step 2: 逐条润色

对每一条进行**适度润色**，使其达到专业正式的职场汇报标准。

**要做的：**
- 口语 → 书面语（"改了个bug" → "完成XX模块的缺陷修复"）
- 补充合理的动词前缀（完成、优化、推进、对接、排查、梳理、落地、上线）
- 每条是一个完整的「动作 + 成果」描述
- 保留用户输入中的专业术语和具体模块名

**不要做的：**
- 不凭空编造数据、人名、技术细节
- 不过度展开成长篇大论
- 不改变用户的核心意思
- 不使用 emoji、感叹号等非正式元素

**润色示例：**

| 用户输入 | 输出 |
|---------|------|
| 修复登录bug | 排查并修复登录模块的认证缺陷 |
| 改首页 | 优化首页页面加载性能 |
| 对接支付接口 | 完成支付接口的联调对接工作 |
| 开会 | 参与项目进度同步会议，明确下一阶段任务分工 |
| 写文档 | 梳理并输出技术文档 |
| code review | 完成代码评审，反馈优化建议 |

### Step 3: 根据数量调整详略

| 任务数 | 策略 |
|--------|------|
| 1-3 条 | 适当展开，每条可多描述过程或成果 |
| 4-6 条 | 保持适中，一句话说清 |
| 7 条以上 | 简洁精炼，半句到一句话 |

### Step 4: 输出

最终输出为**纯文本编号列表**：

```
1. [润色后的任务描述]
2. [润色后的任务描述]
3. [润色后的任务描述]
```

格式规则：
- 使用 `数字.` 序号（1. 2. 3.）
- 不要标题、日期、分割线、Markdown 格式
- 不要「今日完成」等小标题
- **只输出列表本身**，不附加任何解释
- 不要追问"是否需要修改"

## 完整示例

**用户输入：**
```
修了个支付页面的bug
跟产品对了下需求
优化了下列表查询性能
上线了新版本
```

**输出：**
```
1. 排查并修复支付页面的功能缺陷
2. 与产品团队进行需求评审，明确迭代方向
3. 优化列表查询性能，提升数据加载效率
4. 完成新版本发布上线
```

---

## 特殊情况

- **未换行的多任务**：先拆分再逐条处理
- **极其模糊的输入**（如"干活""上班"）：基于常见工作场景合理推断，但保持通用，不编造细节
- **用户附带额外指令**（如"重点写第一条"）：遵从用户意图调整
